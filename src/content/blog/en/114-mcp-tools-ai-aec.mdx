This post is about what those 114 tools are, what they enable, and why no one else in CAD/PLM has anything remotely close.

## What MCP Is

Model Context Protocol is an open standard for connecting AI assistants to external tools. Think of it as function calling, but standardized -- any MCP-compatible assistant (Claude Desktop, Cursor, Claude Code, and others) can discover and use MCP tools without custom integration code.

The protocol works in three steps:

1. An MCP server starts and declares its available tools
2. An AI assistant connects and reads the tool catalog
3. When a conversation requires live data or real actions, the assistant calls the appropriate tool

RAPS is the only production MCP server for any CAD/PLM platform. Full documentation is at [/docs/mcp-server](/docs/mcp-server). The MCP specification itself lives at [modelcontextprotocol.io](https://modelcontextprotocol.io).

## What This Enables

The best way to understand the impact is to see real conversations.

### Example 1 -- Project Setup

> **You:** Set up a new project called "Hospital Wing C", add the structural team, and create the standard folder structure.

The AI calls `project_create`, then `project_user_add` five times (one per team member), then `folder_create` eight times for the standard directory tree. Total wall time: under 30 seconds. Manual equivalent: 15-20 minutes of clicking through the ACC admin UI.

### Example 2 -- Issue Triage

> **You:** Show me all critical issues in the Hospital project that are overdue.

The AI calls `issue_list` with status and priority filters, formats the results into a table with assignees and due dates. No Postman. No GraphQL playground. No token management.

### Example 3 -- Translation Pipeline

> **You:** Upload this Revit file, translate it to SVF2, and tell me when it's ready.

The AI calls `object_upload` to push the file to OSS, `translate_start` to kick off the Model Derivative job, then polls `translate_status` until the job completes. You get a progress update in plain language.

### Example 4 -- Bulk Admin

> **You:** Remove Sarah from all projects -- she is moving to a different division.

The AI calls `admin_user_remove` with the account ID and Sarah's email. One sentence, one operation, applied across every project in the account. The alternative is navigating to each project individually in the web UI.

### Example 5 -- Quality Check

> **You:** List all open checklists for the Hospital project and summarize which ones are behind schedule.

The AI calls `acc_checklists_list`, filters for open items, and presents a summary with completion percentages and overdue flags.

## Architecture

Starting the MCP server takes one command:

```bash
# Local mode — connects to your local raps installation
raps mcp serve

# Or use the hosted version
# Configure mcp.rapscli.xyz in Claude Desktop, Cursor, or any MCP client
```

Claude Desktop configuration:

```json
&#123;
  "mcpServers": &#123;
    "raps": &#123;
      "command": "raps",
      "args": ["mcp"],
      "env": &#123;
        "APS_CLIENT_ID": "your_client_id",
        "APS_CLIENT_SECRET": "your_client_secret"
      &#125;
    &#125;
  &#125;
&#125;
```

Under the hood:

- **Built on rmcp 1.1** -- the Rust MCP library, compiled into the same binary as the CLI
- **Same raps-kernel core** as the CLI -- identical auth, rate limiting, retry logic, and error handling
- **20 concurrent operations max** -- tuned to respect APS rate limits without tripping 429s
- **Sensitive data scrubbed** from all responses -- no tokens, secrets, or credentials leak to the AI
- **Clients cached for performance** -- created on-demand for specialized APIs, reused across tool calls

The MCP server is not a wrapper around the CLI. It shares the same Rust kernel. Every improvement to the CLI automatically improves the MCP tools, and vice versa.

## Why This Matters for AEC

AEC professionals are not developers. They should not need to be.

The traditional path to APS automation requires learning OAuth 2.0, understanding REST API conventions, managing pagination, handling rate limits, and debugging opaque error codes. That path is a filter -- it excludes the vast majority of people who would benefit from automation.

MCP tools change the economics. A project manager who can describe what they need in plain language now has access to the same automation that previously required a developer. The AI handles the API complexity. The professional handles the engineering decisions.

This is not about replacing developers. Developers will still build production pipelines, CI/CD workflows, and custom integrations. But the 80% of APS interactions that are ad hoc -- checking status, pulling reports, onboarding users, triaging issues -- those can now happen through conversation.

The convergence point is clear: AI assistants + domain-specific tools + production reliability = accessible automation. RAPS is where those three meet for AEC.
