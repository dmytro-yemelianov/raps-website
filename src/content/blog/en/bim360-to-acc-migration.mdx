In 2023, Autodesk announced the mandatory migration from BIM 360 to Autodesk Construction Cloud. No bulk migration tooling. No permission transfer utility. No project-level export. Thousands of companies -- some with hundreds of active projects -- were told to migrate. Manually.

This is the story of what that actually means for the people on the ground.

---

## The Scale of the Problem

BIM 360 was the construction industry's default cloud platform for years. Document management, issue tracking, RFIs, submittals, model coordination -- it all lived in BIM 360. Entire enterprises built their workflows around it.

Then Autodesk replaced it with ACC: a different architecture, different APIs, a different permission model. And they made migration mandatory. BIM 360 features were deprecated on a rolling basis. There was no "stay where you are" option.

For a small firm with five projects, this was an inconvenience. For an enterprise customer with hundreds or thousands of projects, each containing users, permissions, folder structures, documents, issues, and RFIs -- this was a crisis.

## What Autodesk Provided

Autodesk shipped a web UI that migrates one project at a time. Their documentation recommended contacting support for large migrations. That was it.

Under the hood, the Admin API technically supports both platforms -- but with different endpoints, different authentication flows, and different data models. The HQ v1 API (BIM 360) and the Construction Admin v1 API (ACC) use different field names for the same concepts. Even the account ID format changed: BIM 360 used a plain UUID, ACC wraps it in a `b.` prefix with base64 encoding.

## What Actually Breaks in Migration

This is where the reality sets in. Migration is not just "move files from A to B." It is a cascade of incompatibilities across every layer of the platform.

## The RCW Problem

Revit Cloud Worksharing is the hardest piece of the migration puzzle. Revit models hosted in BIM 360 use cloud worksharing -- live, multi-user collaboration on a single model file. These are not static documents. They are active working files with multiple architects and engineers editing simultaneously.

Migrating an RCW model means:

- All active users must stop working
- The model must be fully synchronized
- Cloud worksharing must be re-established on the ACC side
- All user permissions must be re-mapped
- Worksharing bookmarks and links must be updated

An Autodesk engineer identified up to 5 enterprise clients who needed bulk RCW migration. No tooling exists for it. Each client had to work with Autodesk support on a case-by-case basis.

For everyone else, it was manual. Model by model. Project by project.

## How RAPS Handles Both Platforms

RAPS was built for exactly this kind of problem. The `raps-acc` crate contains both the BIM 360 HQ v1 and ACC Construction Admin v1 endpoint implementations. The CLI auto-detects which platform a project lives on and uses the correct API.

```bash
# RAPS auto-detects BIM 360 vs ACC and uses the right API
raps project list --hub-id "b.account-id"

# Works for both BIM 360 HQ v1 and ACC Construction Admin v1
raps project create --name "New Project" --hub-id "b.account-id"

# Bulk user operations across mixed BIM 360 + ACC projects
raps admin user add "$ACCOUNT" "user@company.com" --role project_admin

# Export permissions from BIM 360, import to ACC
raps admin folder set-permissions "$ACCOUNT" --project "$OLD" --export perms.csv
raps admin folder set-permissions "$ACCOUNT" --project "$NEW" --import perms.csv
```

The key design decision: **same command interface regardless of platform version.** A script that works against BIM 360 today works against ACC tomorrow without modification.

## The Broader Lesson

Forced migrations without tooling are vendor-centric decisions. The vendor saves development cost by not building bulk migration utilities. The customer pays for that savings in manual labor -- hundreds of hours of clicking, re-configuring, and re-testing.

This is a pattern that repeats across every major platform transition:
